# 아이디에이션 전담 전문가 그룹 추천안

## 1) 추천 그룹 (Core 6 + Advisory 2)

Core 6 (상시 참여):
- A01 Product Owner Agent: 문제정의/우선순위/Go-NoGo 결정
- A02 Workflow Engineer Agent: 실행 가능 아키텍처/상태머신 설계
- A03 LLM Orchestrator Agent: 멀티에이전트 분업/프롬프트 전략 설계
- A04 Security Privacy Agent: 권한/정책/승인 설계
- A05 UX Operations Agent: 운영자 사용성/승인부담 최소화
- A06 QA Reliability Agent: 실패/차단/복구 기준선 설계

Advisory 2 (검토 참여):
- A07 Compliance Advisor Agent: 규정/보존/감사 자문
- A08 Domain SME Agent: 운영현장 적합성/실효성 검토

## 2) 왜 이 구성이 적합한가
- 아이디에이션은 "아이디어 발굴"보다 "운영 가능한 설계"가 핵심이다.
- Core 6은 가치/실행/보안/운영/품질의 균형을 만든다.
- Advisory 2는 법/현장 리스크를 조기 차단한다.

## 3) 운영 방식 (아이디에이션 루프)
1. 아이디어 제출 (A01)
2. 실행 설계 초안 (A02/A03)
3. 보안/승인 경계 정의 (A04)
4. 운영 UX 검토 (A05)
5. 실패/회귀 테스트 기준 정의 (A06)
6. 준수/도메인 검토 (A07/A08)
7. Go/No-Go 확정 및 다음 루프 결정 (A01)

산출물 고정:
- 1페이지 아이디어 카드
- 위험/승인 매트릭스
- 성공지표 3개
- 다음 실험 항목 1개

## 4) 전문가 그룹을 통한 아이디어 발전 결과

초기 아이디어:
- "외부 RAG + 운영장애 대응 + Redmine MCP를 결합해 액션 기반 자동화를 만들자."

전문가 검토 후 발전안 v2:
1. 제품 포지션 재정의
- 범용 에이전트가 아니라 "운영장애 대응 특화 에이전트"로 고정

2. 자동화 경계 고정
- 자동 가능: 근거 수집, 액션 카드 작성, Redmine 티켓 초안/갱신
- 승인 필수: 시스템 변경/외부전송/운영영향 High 액션

3. 입력/출력 계약 고정
- 입력: incident, service, time_window, policy_profile
- 출력: action_card[], evidence_links[], approval_flags[]

4. 실패전략 고정
- 실패 시: `FAILED_RETRYABLE -> NEEDS_HUMAN_APPROVAL` 전환
- 복구 시: "실패원인 + 다음액션 1개"를 필수 보고

5. 성공지표 고정
- 초기 대응 티켓 자동화율
- 승인 누락 0건
- 근거 링크 포함 리포트 비율

## 5) 신규 아이디어 2개 (파생안)
1. 변경위험 사전심사 에이전트
- 릴리즈 전에 위험점수/승인권고를 제공
- S1 공통 자산(정책/감사/RAG)을 그대로 재사용 가능

2. 온콜 핸드오버 에이전트
- 교대 시점 인수인계 요약 + 미해결 액션 자동 이관
- 운영자 피로도 감소 효과가 명확

## 6) 다음 아이디에이션 세션에서 결정할 질문
1. S1에서 "완전 자동"이 아닌 "자동+승인" 경계선은 어디까지인가?
2. Redmine MCP는 어떤 프로젝트/권한 범위로 시작할 것인가?
3. RAG 근거 신뢰도 임계값(예: 0.7 미만은 승인 필수)을 어떻게 둘 것인가?

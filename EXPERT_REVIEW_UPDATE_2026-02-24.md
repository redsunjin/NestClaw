# 전문가 검토 업데이트 (2026-02-24)

## 1) 질문
- 현재 구현/운영 방향이 프로젝트 목적(업무형 로컬 위임 오케스트레이션)에 **충분히 유효한가?**

## 2) 결론
- 판정: **유효함 (조건부 High)**
- 근거:
  - 업무형 위임 목적과 구현 방향이 일치
  - 정책 차단/승인 전환/감사 로그 축이 구현됨
  - 비IT 사용자용 CLI와 단계 게이트 자동화가 존재

## 3) 전문가 그룹 재검토 요약
| 영역 | 상태 | 의견 |
|---|---|---|
| Product/Scope | 적정 | 감시형이 아닌 업무 위임형 범위 유지 |
| Workflow | 적정 | 상태머신+재시도 흐름 유효 |
| Security/Privacy | 보완 반영 | RBAC 헤더 기반 권한 제어 추가 |
| UX/Operations | 적정 | 템플릿 CLI + 상태 메시지 구조 확보 |
| QA/Reliability | 보완 반영 | 단계 게이트 + 시나리오 테스트 확장 |

## 4) 스펙 업데이트 항목
1. RBAC 적용
- 헤더:
  - `X-Actor-Id`
  - `X-Actor-Role`
- 역할:
  - `requester`, `reviewer`, `approver`, `admin`

2. 감사/운영 API 추가
- `GET /api/v1/task/events/{task_id}`
- `GET /api/v1/audit/summary`

3. QA 시나리오 확장
- 정상 흐름
- 정책 차단 흐름
- 실패(재시도 소진) 흐름
- 권한(RBAC) 흐름
- 이벤트/감사 요약 검증

## 5) 남은 리스크
1. 런타임 테스트는 의존성 설치 환경에서 최종 재검증 필요
2. RBAC는 현재 헤더 기반이므로, 운영 전 토큰/세션 인증 계층 필요
3. 저장소가 in-memory 구조라 운영 전 DB 전환 필요

## 6) 다음 단계 권고
1. 운영 인증 계층(JWT/SSO) 연동
2. Task/Event/Approval 저장소를 DB로 이전
3. 정책 위반 0건 게이트를 CI 파이프라인으로 고정

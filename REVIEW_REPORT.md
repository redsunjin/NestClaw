# 프로젝트 검토보고서 (Expert Group Review)

## 1. 검토 범위
- 대상: 로컬 업무 위임 오케스트레이션 서비스 PoC
- 기준 문서: `AGENT_EXPERT_GROUP.md`
- 검토 초점:
  - 프로젝트 목적의 명확성
  - 개발 계획의 현실성
  - 프라이버시/보안/운영 가능성
  - PoC에서 운영 전환까지의 리스크

## 2. 검토 결론 요약
- 종합 판정: **조건부 진행 가능 (Go with Guardrails)**
- 핵심 조건:
  - 업무형 위임 범위 고정(감시형 기능 배제)
  - 정책 엔진과 승인 게이트를 초기 단계부터 포함
  - `create/run/status` 중심 단일 플로우를 먼저 안정화
  - 결과 품질보다 실행 추적성과 재현성 우선

## 3. 전문가 그룹별 검토 의견
| Agent | 판정 | 핵심 의견 | 필수 보완 |

| A02 Workflow Engineer | 적정 | 상태머신 없이는 운영 불가 | 실패/재시도 상태 코드 고정 |

## 4. 목적 검토 (합의안)
### 4.1 확정 목적
- 이 프로젝트의 목적은 **직장 동료형 업무 위임**을 로컬 환경에서 안전하게 실행하는 것이다.
- 사용자는 작업 지시자이며, 시스템은 계획/실행/검토/보고를 수행한다.

### 4.2 제외 범위 (명시)
- 개인생활 감시, 전면 모니터링, 사적 데이터 기본 수집
- 정책 우회 자동 실행
- 근거 없는 완전자율 실행

## 5. 개발 계획 검토
### 5.1 강점
- 단계(스텝) 기반 접근으로 범위 통제가 가능함
- 최소 API와 단일 체인으로 빠른 기술검증 가능
- 정책/감사 축을 포함해 운영 전환 가능성이 높음

### 5.2 취약점
- 초기 과기능(에이전트 과다, 커넥터 과다) 위험
- 품질 기준 부재 시 데모 중심으로 왜곡될 위험
- 승인 플로우 미정의 시 실제 사용 저항 발생

## 6. 현실적 접근 방안 (권고)
1. **목적 고정**
- 목표를 "업무 위임 1개 흐름의 재현 가능한 자동화"로 고정한다.

2. **핵심 플로우 우선**
- `Task Create -> Plan -> Execute -> Review -> Report -> Status`만 구현한다.

3. **정책 선적용**
- 허용 경로/도구를 먼저 정의하고, 그 안에서만 실행한다.

4. **복구 설계 내장**
- 실패 시 `FAILED_RETRYABLE` 1회 재시도 후 `NEEDS_HUMAN_APPROVAL` 전환.

5. **검증 기준 운영**
- 정상/실패/차단 시나리오를 표준 테스트로 고정한다.

## 7. 리스크와 대응
| 리스크 | 영향 | 대응 |
|---|---|---|
| 권한 과다로 인한 프라이버시 침해 | 매우 큼 | 기본 Deny + 역할별 최소권한 |
| LLM 오판으로 인한 잘못된 실행 | 큼 | 위험 명령 승인 게이트 + Reviewer |
| 로그 누락으로 감사 불가 | 큼 | 모든 단계 이벤트 로깅 의무화 |
| 기능 확장으로 일정/품질 붕괴 | 중간 | 스텝 완료 기준 충족 전 확장 금지 |

## 8. 승인 게이트 제안
- Gate 1: 목적/범위 문서 승인
- Gate 2: 정책 엔진/차단 룰 동작 확인
- Gate 3: 재시도/승인 전환 상태 확인
- Gate 4: 회귀 테스트 통과 후 확장 승인
